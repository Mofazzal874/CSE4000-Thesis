## 2. WHY FURTHER MAMBA/SSM BACKBONE CHANGES ARE RISKY

### The Fundamental Problem with Mamba in Detection

Mamba SSMs process sequences in a specific scan order (typically raster scan). For 2D images, this means:
- **Spatial locality is broken** - adjacent pixels in 2D are far apart in 1D scan
- **Bidirectional context is limited** - unlike self-attention which sees all tokens simultaneously
- **Small objects get diluted** - SSM state gets overwritten by background pixels during scanning

This explains exactly why your Mamba model helped overall mAP (better global context for medium objects) but hurt very-tiny recall (small object features get washed out in the state).

### MambaVision: Not What You Think It Is

NVIDIA's MambaVision (CVPR 2025) is interesting but problematic for your case:

- **Smallest variant (Tiny): 31.8M params, 4.4G FLOPs** - this is just the backbone, before detection head
- **Detection results use Cascade Mask R-CNN** (a very heavy two-stage detector), not YOLO
- **Their COCO mAP: 51.1** is with Cascade Mask R-CNN 3x schedule - completely different setup
- **License: CC-BY-NC-SA-4.0** (non-commercial only)
- **Integration with YOLO: non-trivial** - MambaVision outputs 4-stage hierarchical features that don't directly map to YOLO's neck

To use MambaVision as a YOLO backbone, you would need to:
1. Extract the 4-stage feature maps and adapt channel dimensions
2. Redesign the FPN/PANet neck to accept these features  
3. Retrain from scratch (no pretrained detection weights for YOLO)
4. Handle 31.8M+ params on Kaggle P100/T4 with 16GB VRAM

**Risk: HIGH. Timeline: 3-4 weeks minimum. Probability of improvement: LOW.**

### Mamba YOLO (AAAI 2025): A Better Option But Still Risky

The published Mamba YOLO uses ODMamba backbone with RG Blocks. Their tiny variant claims 7.5% mAP improvement with 1.5ms inference on RTX 4090. However:
- Results are on COCO, not aerial/small-object datasets
- You'd be using someone else's architecture (less novelty for your thesis)
- Adapting to your C2A dataset + P2 head requires significant work
- 1.5ms on 4090 ≠ similar speed on P100/T4

### DIRECTION A: Polish What You Have + SAHI (LOW RISK, HIGH REWARD)

- Comprehensive ablation study (baseline, CBAM, P2, Mamba variants)

**What to add:**
1. **SAHI (Slicing Aided Hyper Inference)** - This is an inference-time technique that slices large aerial images into overlapping patches, runs detection on each, then merges results. It specifically targets tiny objects in aerial imagery.
   - No retraining needed
   - Can be applied to your best model
   - Proven to improve small object recall by 5-15% in aerial datasets
   - Paper: "Slicing Aided Hyper Inference and Fine-tuning for Small Object Detection" (2022)
   - Python package: `sahi` (pip install sahi)

2. **Test-Time Augmentation (TTA)** - Multi-scale inference + horizontal flip
   - Free performance boost at inference time
   - 2-3% mAP improvement typical

3. **Copy-Paste Augmentation** - During training, copy small human instances and paste them at random locations
   - Specifically helps small object detection
   - Can retrain your best model with this augmentation
   - Built into ultralytics YOLO with `copy_paste` parameter

**Why this direction:** You get measurable improvements on the exact metric that matters (tiny human recall) without risky architecture changes. The thesis story becomes: "We systematically explored SSM integration, identified its limitations for tiny objects, and proposed complementary inference strategies."

### DIRECTION B: Neck/Feature Fusion Improvements (MEDIUM RISK, MEDIUM REWARD)
**Recommended as Priority 2 if time permits**

Instead of touching the backbone, improve how multi-scale features are fused:

1. **BiFPN (Bidirectional Feature Pyramid Network)** from EfficientDet
   - Weighted bidirectional cross-scale connections
   - Better than PANet for small objects
   - Multiple papers show +1-2% mAP on aerial datasets
   - Relatively easy to implement in ultralytics

2. **AFPN (Asymptotic Feature Pyramid Network)**
   - Recent work showing better small object fusion
   - Lower compute overhead than full BiFPN

3. **Gold-YOLO's GD (Gather-and-Distribute) mechanism**
   - Better information flow between scales
   - Specifically designed for YOLO architecture

**Why this direction:** Neck improvements have historically been more reliable than backbone swaps for detection tasks. Your P2 head already proves that multi-scale handling is where the gains are.

### DIRECTION C: Lightweight Hybrid CNN-Mamba in Neck Only (MEDIUM-HIGH RISK, HIGH NOVELTY)
**Only if you need more novelty for thesis**

Instead of replacing backbone layers with Mamba (which you've shown hurts tiny objects), use a small Mamba module ONLY in the neck/feature fusion stage:

- Keep CNN backbone (proven to work)
- Keep P2 head (your best contributor)
- Add a lightweight SSM block that processes cross-scale feature sequences
- The SSM would model dependencies ACROSS scales rather than within spatial dimensions

This is genuinely novel because:
- Published Mamba YOLO variants all modify the backbone
- Using SSM for cross-scale temporal-like modeling in the neck is unexplored
- It addresses the actual weakness (multi-scale fusion) rather than replacing what already works (CNN features)

**Risk:** Medium-high because it's untested, but the scope is small (one module in the neck) so failure is cheap.
